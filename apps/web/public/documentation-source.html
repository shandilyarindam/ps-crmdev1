End-to-End Complaint Journey

One complaint, traced from citizen's mouth to proof photo — every system that fires along the way.

Seema reports a sewage overflow on her street in Delhi.

  1. 01
    Intake via Seva AI Seema opens JanSamadhan, types in Hindi: "Hamare gali mein nali overflow ho rahi hai — bahut gandagi hai." Gemini classifies in seconds: Category → Sewer Line Blockage · Authority → DJB · Severity → L3 High (48h SLA). Seva shows a preview — Seema confirms.
  2. 02
    Location Pinning + DIGIPIN GPS auto-detects her location. She drags the map pin 5m to the exact overflow spot. DIGIPIN encoded: 39J-948-H2PK (4m×4m square). PostGIS 20m scan runs — no duplicate found. Complaint proceeds.
  3. 03
    Ticket Created Turnstile verification passes. Ticket inserted: DL-2026-00312. SLA deadline: 48 hours from now. Email + WhatsApp confirmation sent with deep-link to her ticket UUID. Ticket history row created (submitted state).
  4. 04
    Authority Receives + Assigns DJB authority logs in. RLS ensures they see only DJB complaints. DL-2026-00312 appears at top (L3 High). They assign to field worker Rajan. The assign_worker_to_complaint RPC fires with a FOR UPDATE row lock — no race condition possible.
  5. 05
    Worker Field Execution Rajan sees the ticket on his mobile dashboard. He taps Navigate — Google Maps opens at the DIGIPIN coordinates. He marks In Progress. Clears the drain. Uploads a proof photo to complaints-media/resolution/. Status → Resolved.
  6. 06
    DB Trigger Fires Automatically On resolution, a PostgreSQL trigger fires: Rajan's availability resets to Available, current_complaint → NULL, resolved_at timestamp set. No API call required — the database handles state cleanup.
  7. 07
    Citizen Confirmation + Rating Seema gets a WhatsApp message with a deep-link. She sees the proof photo. She confirms resolved. Rating window opens — she rates 8/10. Wallet points awarded to both Seema and Rajan.
  8. 08
    Immutable Audit Trail Full ticket_history shows every state change with timestamps. Visible to admin, partially to citizen, internal authority notes hidden. Tamper-proof. Total time: 6 hours against a 48-hour SLA.
Result: Resolved in 6 hours against a 48h SLA. Full digital trail. Proof photo on record. Citizen rated it. Worker rewarded. Model gets smarter from the event.
05

Trust & Accountability Guarantees

These are not UI rules. They are database-level constraints that cannot be bypassed by any user, including admins.
  • ✓
    No duplicate ticket for the same issue within 20m
    Enforced via PostGIS ST_DWithin before INSERT. The check runs at DB layer, not application layer. RPC: check_for_duplicate_report
  • ✓
    No invalid status transition is possible
    A BEFORE UPDATE trigger raises a PostgreSQL exception on any invalid state change. Bypassing the UI does not help. Trigger: enforce_status_transition()
  • ✓
    No authority can access another department's complaints
    28 RLS policies use JWT claims at the DB layer — not application logic. Even direct DB queries from outside the app are blocked. 28 RLS Policies
  • ✓
    No race condition on worker assignment
    FOR UPDATE row lock inside the assignment RPC prevents two authorities assigning the same worker simultaneously. RPC: assign_worker_to_complaint
  • ✓
    Ticket IDs are globally unique
    PostgreSQL SEQUENCE guarantees. DL-YYYY-NNNNN never repeats even under concurrent submissions.
  • ✓
    SLA breach detection is fully automatic
    Cron-callable check_sla_breaches() RPC requires no application uptime to escalate past-deadline complaints.
  • ✓
    Severity never decreases
    effective_severity = max(severity, upvote_boost) enforced inside the upvote RPC. Once escalated, it stays.
  • ✓
    DIGIPIN encoding works without internet
    Pure algorithm — no external API call at encoding time. Works in low-connectivity areas and offline.
These guarantees hold even if the frontend crashes, the backend is bypassed, or a user has DB access. The database is the last line of defence — and it holds.

📍 DIGIPIN & Location Intelligence

DIGIPIN (Digital Postal Index Number) is India's official location coding system by the Department of Posts, GoI.

GPS coordinate → FastAPI /digipin/:lat/:lng → 10-char code Example: 28.6315°N, 77.2167°E → 39J-948-H2PK (4m × 4m area)
Why DIGIPIN MattersImpact
Works without street namesSlums, villages, new colonies — no address needed
Offline capablePure algorithm — no API call, works on 2G
4m × 4m precisionWorkers navigate to exact spot, not a 200m street
Survives renamingPermanent identifier despite address/ward changes
Ward-level aggregatesAnalytics by DIGIPIN prefix — cluster intelligence
Duplicate anchorPostGIS ST_DWithin on geography column uses it
🔍 Duplicate Detection & Community Upvote Engine
Before submission: check_for_duplicate_report(lat, lng, category_id) PostGIS ST_DWithin → 20m radius, same category, active status ↓ Duplicate found → show existing ticket → citizen upvotes instead ↓ No duplicate → submission proceeds increment_upvote_count(complaint_id) RPC → Atomic: upvote count + severity recalculation in one transaction → effective_severity = max(base_severity, upvote_tier) → NEVER decreases At 15+ upvotes: possible_duplicate flag raised for admin review At 50+ upvotes: force L4 Critical regardless of original severity
⏱️ SLA & Auto-Escalation Engine

SLA deadlines set at ticket creation. A PostgreSQL cron-callable RPC handles auto-escalation — no application logic, no uptime dependency.

-- Cron-callable RPC
check_sla_breaches()
  → finds: NOW() > sla_deadline AND status IN (assigned, in_progress)
  → updates: status → 'escalated'
  → surfaces: at top of authority dashboard with ⚠️ indicator

The enforce_status_transition() trigger runs BEFORE every status UPDATE and raises a PostgreSQL exception on any invalid transition.

📹 CCTV / YOLO Auto-Detection System
CCTV feed / dashcam input ↓ AI service: /cctv/analyze_live 10-frame burst → YOLO v8 inference per frame Reliability engine: multi-trigger threshold on frame consensus (prevents single-frame false positives from creating tickets) ↓ Confidence above threshold? ├── Yes → auto-create ticket with DIGIPIN from camera metadata └── No → discard, log event for supervised learning dataset ↓ Admin dashcam-live: /admin/dashcam-live Real-time telemetry cards for approved feeds ↓ Verification: /cctv/verify Admin confirms: repaired | not_repaired Worker event captured → retraining dataset pipeline

Precomputed dashcam artifact endpoints allow demo reproducibility without requiring live camera feeds.

Security & Abuse Prevention

A civic platform with fake complaints is worse than no platform. Every layer has a defence.

RLS role access matrix — what each role can and cannot see.

RoleAccess Scope
👤 CitizenOwn complaints (insert + read), public listing, upvotes, ratings
👷 WorkerOnly their assigned complaints — zero visibility into others'
🏛️ AuthorityOnly complaints from their assigned department — JWT-enforced
⚙️ AdminFull access across all tables and all departments
service_roleBypass for server-side triggers and cron only
  • ✓
    28 RLS policies on all tables — auth.uid() and auth.jwt() based, no recursive subqueries
  • ✓
    ENUM State Machine — invalid values rejected at DB before any trigger or policy runs
  • ✓
    enforce_status_transition() trigger — BEFORE UPDATE raises exception on invalid state. UI bypass does not help.
  • ✓
    Cloudflare Turnstile on complaint submission and login — prevents bot flooding
  • ✓
    FOR UPDATE row locks in assignment RPC — no concurrent assignment race conditions
  • ✓
    DIGIPIN is offline-safe — pure algorithm, no external API call at encoding time
  • ✓
    GCP Secret Manager — no sensitive keys in Docker images, all injected at runtime
  • ✓
    Spam moderation queue — admin flags + wallet penalty + strike system deters abuse
Security here is structural — it works because the database enforces it, not because we trust users or the UI to behave correctly.

WhatsApp deep-link 404Verify FRONTEND_BASE_URL set to deployed domain; URL must use complaint UUID not display code
